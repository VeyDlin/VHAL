# First Project

This guide walks through creating a minimal VHAL project with RTOS, using the STM32G0 demo as a reference. You can find complete demo projects in the `.demo/` directory and use them as a starting point.

## Project Structure

A typical VHAL project looks like this:

```
MyProject/
├── BSP/
│   ├── VHALConfig.h      # Platform and peripheral defines
│   ├── BSP.h              # Pin mapping, adapter declarations
│   ├── BSP.cpp            # Adapter instances, clock and peripheral init
│   └── IRQ/               # Interrupt handlers
├── Application/
│   ├── Application.h      # Task declarations, RTOS startup
│   ├── Application.cpp    # Task instances, thread creation
│   └── Tasks/             # Task classes (one per file)
├── main.cpp               # Entry point
├── CMakeLists.txt
└── CMakePresets.json
```

## VHALConfig.h

This file selects the platform and enables the peripherals your project uses. VHAL reads these defines to include the correct headers and port adapters automatically.

```cpp
#pragma once

// Platform
#define VHAL_STM32
#define VHAL_STM32_G0

// Peripherals
#define VHAL_STM32_G0_UART

// System
#define VHAL_RUNTIME
#define VHAL_RTOS
#define VHAL_RTOS_FREERTOS
```

`VHAL_RUNTIME` enables built-in C++ runtime stubs (`new`/`delete`, `syscalls`, `sbrk`) for bare-metal platforms that don't provide their own. `VHAL_RTOS` and `VHAL_RTOS_FREERTOS` enable the OS abstraction layer.

See the [VHAL Config](/docs/VHAL/VHAL_Config) page for the full list of available defines.

## BSP

BSP (Board Support Package) declares hardware adapters and handles low-level initialization: clocks, pins, interrupts.

### BSP.h

```cpp
#pragma once
#include <VHAL.h>

class BSP {
public:
    static AUART consoleSerial;
    static AGPIO ledPin;

public:
    static void Init() {
        InitSystem();
        InitClock();
        InitSystemTick(1, 0);
        System::Init();
        InitAdapterPeripheryEvents();
        InitIO();
    }

private:
    static void InitSystem();
    static void InitClock();
    static void InitSystemTick(uint32 ms, uint32 tickPriority);
    static void InitAdapterPeripheryEvents();
    static void InitIO();
};
```

Adapter types like `AUART`, `AGPIO` are aliases provided by VHAL for the selected platform. You declare them as static members and initialize in `BSP.cpp`.

### BSP.cpp

```cpp
#include "BSP.h"

AUART BSP::consoleSerial = { USART1 };
AGPIO BSP::ledPin        = { GPIOC, 6 };


void BSP::InitClock() {
    // Configure system clock (platform-specific)
    // See demo projects for complete examples
}

void BSP::InitSystemTick(uint32 ms, uint32 tickPriority) {
    SysTick_Config(System::GetCoreClock() / (1000U / ((1000U * ms) / 1000U)));
    NVIC_SetPriority(SysTick_IRQn, tickPriority);
}

void BSP::InitAdapterPeripheryEvents() {
    consoleSerial.beforePeripheryInit = []() {
        // Enable peripheral clock
        LL_APB2_GRP1_EnableClock(LL_APB2_GRP1_PERIPH_USART1);

        // Configure GPIO alternate function for UART pins
        AGPIO::AlternateInit({ GPIOA, 9,  1, AGPIO::Pull::Up, AGPIO::Speed::VeryHigh }); // TX
        AGPIO::AlternateInit({ GPIOA, 10, 1, AGPIO::Pull::Up, AGPIO::Speed::VeryHigh }); // RX

        // Enable interrupt
        NVIC_SetPriority(USART1_IRQn, 0);
        NVIC_EnableIRQ(USART1_IRQn);

        return ResultStatus::ok;
    };
}

void BSP::InitIO() {
    ledPin.SetParameters({ AGPIO::Mode::Output });
}
```

VHAL adapters handle peripheral register configuration (baud rate, mode, data format, etc.), but they intentionally do not manage things that are tied to the board design rather than the peripheral itself — pin routing (alternate functions), peripheral clock gating, and interrupt priorities. These depend on your specific PCB and cannot be abstracted.

Every adapter provides two callbacks:
- `beforePeripheryInit` — runs before the adapter configures peripheral registers. Use it for clock gating, pin alternate functions, and NVIC setup.
- `afterPeripheryInit` — runs after the adapter finishes configuration. Use it for any post-init steps like enabling DMA or additional pin setup.

Both return `ResultStatus` and are called automatically when you call `SetParameters()` on the adapter.

## Creating a Task

Tasks are classes that inherit from `ThreadStatic<StackSize>` and override the `Execute()` method. The template parameter sets the stack size in words.

```cpp
// Application/Tasks/MainTask.h
#pragma once
#include <Application.h>

class MainTask: public ThreadStatic<256> {
public:
    void Execute() override {
        BSP::consoleSerial.SetParameters({
            .baudRate = 115200
        });

        BSP::consoleSerial.WriteString("Hello from VHAL!\r\n");

        while (true) {
            BSP::ledPin.Toggle();
            Sleep(500ms);
        }
    }
};
```

Key points:
- `Execute()` is the task entry point, like a `main()` for the thread
- `Sleep()` yields the CPU to other tasks for the specified duration
- Use C++ chrono literals: `500ms`, `1s`, `10ms`
- Access hardware through `BSP::` static members

## Application

Application creates task instances, registers them with the RTOS, and starts the scheduler.

### Application.h

```cpp
#pragma once
#include <BSP.h>

using namespace OS;

class Application {
public:
    static class MainTask main;

public:
    static void Init();
};
```

### Application.cpp

```cpp
#include <Application.h>
#include <Tasks/MainTask.h>

MainTask Application::main;

void Application::Init() {
    RTOS::CreateThread(main, ThreadPriority::normal, "main");
    RTOS::Start();
    // Code after RTOS::Start() never executes
}
```

`RTOS::CreateThread()` takes a thread reference, optional priority, and optional name for debugging. `RTOS::Start()` starts the scheduler — it never returns.

## main.cpp

The entry point is minimal:

```cpp
#include <Application.h>

int main(void) {
    BSP::Init();
    Application::Init();
}
```

## Thread API

| Method | Description |
|--------|-------------|
| `Sleep(duration)` | Suspend the current thread for a duration (`100ms`, `1s`) |
| `SleepUntil(duration)` | Sleep until next period (for periodic tasks) |
| `Signal(mask)` | Send a signal to this thread from another thread or ISR |
| `WaitForSignal(timeout, mask)` | Block until a signal is received or timeout |
| `Suspend()` | Suspend the thread |
| `Resume()` | Resume a suspended thread |

## UART Receive Example

A more complete example — LED toggles on UART receive using signals:

```cpp
class UartTask: public ThreadStatic<256> {
    void Execute() override {
        BSP::ledPin.SetParameters({ .mode = AGPIO::Mode::Output });

        uint8 rxBuffer[64];
        BSP::consoleSerial.SetParameters({ .baudRate = 115200 });
        BSP::consoleSerial.onInterrupt = [this](AUART::Irq irq) {
            if (irq == AUART::Irq::Rx) {
                Signal();
            }
        };
        BSP::consoleSerial.SetContinuousAsyncRxMode(true);
        BSP::consoleSerial.ReadArrayAsync(rxBuffer, sizeof(rxBuffer));

        while (true) {
            WaitForSignal();
            BSP::ledPin.Toggle();
        }
    }
};
```

The task sleeps on `WaitForSignal()` until the UART interrupt fires and calls `Signal()`, waking the task to process the data.
